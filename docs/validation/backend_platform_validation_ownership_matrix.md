# Backend and Platform Validation Ownership Matrix

> This matrix closes the non-browser portion of the product workflow inventory by assigning backend, platform, and release-gate ownership for evidence that cannot be proven by frontend journey tests alone.

## Matrix Maintenance Contract

This document is the canonical ownership map for **non-browser release evidence**. It complements the release inventory in [`docs/validation/master_workflow_traceability_matrix.md`](master_workflow_traceability_matrix.md), the frontend subset in [`apps/web/docs/frontend-workflow-coverage-matrix.md`](../../apps/web/docs/frontend-workflow-coverage-matrix.md), the backend object lineage contract in [`docs/contracts/workflow-traceability-matrix.md`](../contracts/workflow-traceability-matrix.md), and the backend-integrated milestone matrix in [`docs/validation/backend_integrated/backend_integrated_traceability_matrix.md`](backend_integrated/backend_integrated_traceability_matrix.md).

The matrix must fail closed in CI through `python3 scripts/ci/assert_backend_platform_validation_ownership.py`, which is invoked by the root `make check-workflow-matrix` target. New release-significant backend or platform workflows must update this document whenever they introduce new authentication policy, persistence semantics, audit guarantees, observability surfaces, retention controls, or external-system integration behavior.

## Gate Integration

| Gate | Scope | Required Evidence | Failure Interpretation |
|---|---|---|---|
| `make check-workflow-matrix` | Documentation traceability | Master workflow matrix, backend object lineage, and this backend/platform ownership matrix all retain release-critical markers. | A missing marker means release evidence ownership was dropped and must be restored before merge. |
| `make test-backend-integrated-validation` | Live backend-integrated proof | Golden path, cross-layer flow, tenant persistence, agent grounding, calculation provenance, approval/export/CRM governance, and operational resilience. | A failing suite means the workflow remains unproven outside the browser and must not be treated as release-ready. |
| `make test-backend-integrated-release-smoke` | Release-candidate smoke | Health, auth, account creation, ingestion, calculator, business-case generation, tenant isolation, and export gate checks. | A failing smoke run means the release environment does not have sufficient operational proof for ship review. |
| `pytest services/api/app/tests/test_auth_enforcement.py services/api/app/tests/test_health.py services/api/app/tests/test_i03_durable_persistence_and_llm.py services/api/app/tests/test_production_safety.py` | Direct API and configuration proof | Token enforcement, tenant mismatch denial, public health/metrics, durable persistence behavior, and production fail-closed policy. | A failing direct test means core API/platform policy is broken even before integrated service proof is attempted. |

## Open Implementation Backlog

The ownership gap is now documented and guarded, but several evidence items remain implementation backlog rather than proven passing behavior. The next backend/platform validation batch should add **release IdP SSO/MFA smoke**, **retention and deletion policy tests**, **restart-safe ingestion job recovery checks**, and **append-only audit immutability assertions**. Those tests should be added as fail-first suites and promoted into `make test-backend-integrated-validation` only after they exercise live contracts rather than static documentation.
