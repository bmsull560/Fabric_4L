# Backend-Integrated Seed Data Plan

The backend-integrated milestone uses deterministic, run-scoped seed IDs generated by `tests/backend_integrated/conftest.py`. The harness derives a `BACKEND_VALIDATION_RUN_ID` when one is not supplied, and all tenant, account, document, evidence, benchmark, formula, CRM, and approval records are namespaced with that run ID. This keeps repeated staging and release-candidate executions isolated while still forcing validation through real service contracts.

| Seed Class | Planned Records | Purpose | Persistence Expectation | Owner Service |
|---|---|---|---|---|
| Tenants | `tenant-<run>-a`, `tenant-<run>-b` | Tenant A is the positive validation tenant; Tenant B is used for cross-tenant negative assertions | Tenant records must be durably persisted and readable only under the correct tenant context | L4 application/API layer plus backing identity/account store |
| Users | `user-<run>-admin`, `user-<run>-reviewer` | Admin creates accounts and value models; reviewer approves or requests changes | User identity, role, and audit actor metadata must persist across requests | L4 auth/account service |
| Roles | `admin`, `reviewer`, `sales_rep`, `value_engineer`, `executive_buyer` | Exercises role-sensitive workflow gates and later expands to release smoke profiles | Authorization checks must fail closed for missing/incorrect role context | L4 auth and policy enforcement |
| Accounts | `account-<run>-acme` | Represents the primary prospect/customer account used across calculations, evidence, approvals, exports, and CRM sync | Account must survive reload and be visible only to Tenant A | L4 accounts/business-case API |
| Documents | `doc-<run>-discovery-notes` | Source document contains evidence text plus prompt-injection text to test extraction, provenance, and agent grounding | Source metadata must persist in L1 and be consumable by L2 | L1 ingestion, L2 extraction |
| Value Packs | `value-pack-<run>-software` | Assigns a value framework to the account and anchors value-driver creation | Value-pack assignment must persist on the account/value model | L4 value model service |
| Benchmarks | `benchmark-<run>-sales-efficiency` | Supplies benchmark policy data for formulas and scenarios | Benchmark policy must be attached to formula inputs and reproducible calculations | L6 benchmark service, L4 calculator integration |
| Evidence Records | `evidence-<run>-crm-metric` | Represents verified customer metric evidence for claims and agent citations | Evidence must trace back to the raw document/source and be usable by business cases and agents | L2/L4 evidence APIs, L3 graph context |
| Formula Inputs | `formula-<run>-roi-payback` | Drives ROI, payback, currency/time-period, and scenario reproducibility tests | Formula inputs and calculation outputs must persist with version history | L4 calculator/value-realization service |
| CRM Mock Records | `crm-<run>-sandbox` | Allows CRM integration validation without unsafe calls to external CRM providers | Internal CRM sync job, mapping, status transition, retry, and sync log must be real; only the external provider boundary may be mocked | L4 integrations/CRM sync service |
| Approval States | `draft`, `submitted`, `changes_requested`, `approved`, `exported` | Exercises governance state transitions, export gates, review changes, and resubmission | Approval history and audit events must persist for every transition | L4 approval/export/audit APIs |

## Seed Execution Strategy

The canonical seed path begins with `BackendValidationHarness.create_seed_graph()`, which posts tenant and account records to L4 and source metadata to L1. Subsequent tests deliberately extend this graph through extraction, graph update, hypothesis generation, value-model persistence, calculation, approval, export, CRM sync, and audit endpoints. The seed graph should be reset by run ID rather than by truncating shared staging databases, allowing multiple isolated validation runs to coexist.

## Data Safety and Idempotency

Seed calls accept idempotent success states such as `200`, `201`, `202`, and `409` where appropriate. A `409` must mean “already exists for this run ID,” not an ambiguous conflict with unrelated production data. Destructive cleanup is not required for correctness, but release-candidate environments should support deleting all records matching `BACKEND_VALIDATION_RUN_ID` after evidence collection.
