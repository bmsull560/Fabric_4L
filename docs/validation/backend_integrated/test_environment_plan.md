# Backend-Integrated Test Environment Plan

The backend-integrated milestone is intentionally a **live-service validation gate**. It must fail closed when services, persistence, tenant boundaries, job workers, or audit trails are unavailable. It is not a replacement for the passing frontend E2E validation suite; it is the next additive gate for proving Fabric_4L as a real system of record.

| Environment Requirement | Required Configuration | Validation Rationale |
|---|---|---|
| Required services | L1 ingestion, L2 extraction, L3 knowledge/graph, L4 agents/application APIs, L5 ground-truth validation, L6 benchmarks | The suite validates cross-layer handoffs rather than frontend route mocks. |
| Required ports | Default service URLs are `http://localhost:8001` through `http://localhost:8006` for L1-L6 respectively | The harness reads `LAYER1_API_URL` through `LAYER6_API_URL`, so staging/release URLs may override local defaults. |
| Required environment variables | `BACKEND_VALIDATION_RUN_ID`, `BACKEND_VALIDATION_HTTP_TIMEOUT`, `FABRIC_TENANT_HEADER`, `FABRIC_USER_HEADER`, `FABRIC_ROLE_HEADER`, `LAYER1_API_URL`, `LAYER2_API_URL`, `LAYER3_API_URL`, `LAYER4_API_URL`, `LAYER5_API_URL`, `LAYER6_API_URL` | Run IDs isolate persisted records; service URL variables bind tests to local, staging, or release-candidate deployments; header variables adapt to current auth/tenant propagation contracts. |
| Database reset strategy | Prefer run-scoped cleanup using `BACKEND_VALIDATION_RUN_ID`; avoid truncating shared staging databases | Tests create durable records and then verify reload/persistence. Cleanup should remove only records generated by the validation run. |
| Seed strategy | Create Tenant A, Tenant B, admin, reviewer, account, source document, value pack, benchmark, evidence, formula, CRM mock connection, and approval-state records through real APIs | Seed data must be created through service contracts so tests validate system behavior rather than direct database fixtures. |
| Job worker strategy | L1 ingestion, L2 extraction, graph update, export generation, CRM sync, and long-running job workers must be running or controllably simulated in staging | Operational resilience tests assert queued/running/completed/failed/cancelled states and retryable failures. |
| External dependency mocking strategy | Mock only unsafe external providers such as CRM vendor APIs or external LLM providers; do not mock internal sync jobs, tool-result contracts, persistence, status transitions, audit trails, or calculation behavior | The milestone must prove internal Fabric_4L service behavior with real contracts and durable state. |

## Service Health Expectations

The release smoke suite checks health paths before deeper validation. Accepted health paths are encoded in `tests/backend_integrated/conftest.py`: L1-L3 use `/health` or `/api/v1/health`, L4 uses `/health`, `/v1/health`, or `/v1/health/detailed`, L5 uses `/health` or `/api/v1/health`, and L6 uses `/health` or `/v1/health`.

## Authentication and Tenant Context

The harness sends tenant, user, and role headers on every request. Missing tenant context must fail closed, and cross-tenant access using Tenant B must return an authorization-safe status such as `401`, `403`, or `404`. The release environment may replace header-based validation with token-based authentication later, but the suite should keep explicit tenant-context checks because tenant leakage is a production gate.

## Recommended Commands

The dedicated validation commands are exposed in the repository Makefile. `make test-backend-integrated-validation` runs the full backend-integrated milestone, and `make test-backend-integrated-release-smoke` runs only release-smoke checks. Direct Pytest execution is also supported with `pytest tests/backend_integrated -m backend_integrated`.
