# Fabric_4L Dual-Track Launch Readiness Assessment & Sprint Plan

Use this workflow to assess the current production readiness of the Value Fabric platform using
two lenses:

- **Claimed readiness** from `ROADMAP.md`
- **Verified readiness** from the newest local evidence artifacts, workflows, and test outputs

Historical plans and prior assessments are context only. They are not source of truth for current
launch status.

## Activation Criteria

Manual activation:
- Pre-release readiness reviews
- Quarterly planning sessions
- After major roadmap or infrastructure changes
- Before production deployment or promotion

## Core Principles

1. **`ROADMAP.md` is canonical for claimed completion.**
2. **Local evidence is canonical for verified readiness.**
3. **If claimed and verified disagree, report both and block launch on the verified side.**
4. **Keep the main readiness table aligned to L1-L5, Frontend, and DevOps.**
5. **Report L6 Benchmarks in a separate note, not in the main table.**
6. **Never create or overwrite dated assessment artifacts until the user explicitly approves.**
7. **Treat this repository's older launch-readiness assumptions as stale unless current evidence still supports them.**

## Workflow Steps

1. **Initialize Context**
   // turbo
   - Read `ROADMAP.md` from repository root.
   - Read the newest existing `.windsurf/plans/launch-readiness-*.md` for historical context only.
   - Read `docs/archive/2026-04-19/ARCHIVED_Phase2_Readiness_Status.md` only if historical correction context is needed.
   - Read the newest local evidence artifacts relevant to launch:
     - `artifacts/release/summary.md`
     - `artifacts/release/policy-eval.json`
     - `artifacts/arch/summary.md`
     - `artifacts/arch/report.json`
     - `artifacts/security/summary.md`
     - `artifacts/security/report.json`
     - newest `artifacts/smoke-report-*.json`
     - `test-results/.last-run.json`
     - `build-progress.log`, `e2e-results.log`, and `e2e-test-output.log` when present
   - Read workflow and command sources that define launch verification behavior:
     - `.github/workflows/prod-readiness.yml`
     - `.github/workflows/pr-checks.yml`
     - `.github/workflows/smoke-gate.yml`
     - `Makefile`

2. **Verify Launch-Gate Integrity Before Trusting Any Readiness Claim**
   - Check that every command referenced by `.github/workflows/prod-readiness.yml` exists in the current repo.
   - Check that all referenced policy/config files exist, especially `.fabric/prod-gates.policy.yaml`.
   - Check that expected artifact roots exist or are generated by real workflows:
     - `artifacts/arch/`
     - `artifacts/security/`
     - `artifacts/agent/`
     - `artifacts/state/`
     - `artifacts/obs/`
     - `artifacts/release/`
   - Check that the current baseline verification commands are present and usable:
     - `make gate-arch`
     - `make gate-security`
     - `make gate-state`
     - `make gate-config`
     - `python scripts/smoke/production_smoke.py`
   - If a workflow references missing make targets, missing policy files, or missing artifact paths, record **launch-gate drift** as a P0 blocker and mark affected evidence as **unverified**.

3. **Build a Dual-Track Readiness Table**
   - For each main area, capture:
     - `Claimed`: current percentage/status from `ROADMAP.md`
     - `Verified`: newest evidence-backed value, or `Unverified` / `Blocked`
     - `Target`: launch target for that area
     - `Gap`: shortfall versus target or verification state
     - `Evidence`: short note naming the newest supporting or contradicting artifact
   - Main table rows:
     - L1 Ingestion (target: 90%)
     - L2 Extraction (target: 95%)
     - L3 Knowledge Graph (target: 90%)
     - L4 Agents (target: 85%)
     - L5 Ground Truth (target: 100%)
     - Frontend (target: 85%)
     - DevOps (target: 80%)
   - Report L6 Benchmarks separately as an **L6 Note** with claimed status, verified evidence, and launch relevance.
   - Do not fabricate verified percentages. If the newest evidence is missing, failed, stale, or contradicted by a broken verification path, use `Unverified` or `Blocked`.

4. **Calculate Overall Readiness**
   - Report both:
     - **Claimed Readiness**: based on the current `ROADMAP.md` layer percentages
     - **Verified Readiness**: based on evidence-backed rows only
   - If any launch-critical row is `Blocked` or `Unverified`, explicitly mark overall verified readiness as **Blocked** even if the numeric average is high.
   - Launch-critical blockers include:
     - launch-gate workflow drift
     - security-isolation failures
     - contract drift
     - missing agent/state/observability evidence
     - incomplete monitoring or Kubernetes verification

5. **Identify the Top 5 Launch Blockers**
   Prioritize blockers from current evidence, not from stale assumptions. Default ordering unless fresher evidence overturns it:
   1. Prod-readiness gate drift
   2. Security-isolation failures
   3. Contract drift
   4. Missing agent/state/observability artifacts
   5. Incomplete monitoring / health / Kubernetes verification

   Only reintroduce previously completed roadmap items such as Task 25, 26, 28, 29, or the
   frontend API-wiring work if current evidence shows an actual regression.

6. **Generate a Refreshed 5-Sprint Plan**

   **Sprint 1 â€” Launch Gate Repair and Baseline Evidence**
   - Goal: Align `prod-readiness.yml` to the commands and files that actually exist now.
   - Tasks:
     - [ ] Repair missing policy/config references used by launch gates
     - [ ] Replace or restore missing `make` targets referenced by `prod-readiness.yml`
     - [ ] Ensure launch-gate jobs produce fresh `arch`, `security`, `agent`, `state`, `obs`, and `release` artifacts
     - [ ] Re-run baseline gate sequence using current repo commands
   - Exit Criteria: The launch-gate workflow runs end-to-end and any remaining failures are real product failures, not broken gate plumbing.

   **Sprint 2 â€” Security Isolation and Contract Closure**
   - Goal: Clear the current security-isolation and contract-drift blockers.
   - Tasks:
     - [ ] Fix the reported cross-tenant access success path
     - [ ] Raise critical-endpoint isolation test coverage from 85% to 100%
     - [ ] Clear the single contract-drift violation
     - [ ] Regenerate fresh `artifacts/security/*` and `artifacts/arch/*`
   - Exit Criteria: Fresh arch and security artifacts both pass.

   **Sprint 3 â€” Monitoring, Health, and Kubernetes Verification**
   - Goal: Produce real verification for observability and deployment readiness.
   - Tasks:
     - [ ] Verify Prometheus endpoints return real counters
     - [ ] Verify health checks expose actual dependency status
     - [ ] Verify Kubernetes manifests render cleanly and deploy in staging or equivalent validation path
     - [ ] Produce the missing observability evidence artifacts
   - Exit Criteria: Health, metrics, and Kubernetes checklist items are verified from current evidence, not inferred.

   **Sprint 4 â€” L1 Ingestion Hardening and Runtime Confidence**
   - Goal: Verify or finish the remaining L1 launch-critical runtime work.
   - Tasks:
     - [ ] Confirm actual Celery/Redis wiring status with code and runtime evidence
     - [ ] Complete any remaining rate-limit/runtime hardening needed for launch
     - [ ] Re-run smoke and integration paths against the validated L1 state
     - [ ] Decide whether any remaining L1 gaps are launch blockers or explicit post-launch carryovers
   - Exit Criteria: L1 reaches target with evidence, or is explicitly de-scoped from initial launch with recorded risk acceptance.

   **Sprint 5 â€” Final Evidence Refresh and Go/No-Go**
   - Goal: Re-run the full evidence stack and produce a final launch decision.
   - Tasks:
     - [ ] Re-run gate, smoke, and verification artifacts
     - [ ] Recompute the dual-track readiness table
     - [ ] Refresh the final launch checklist
     - [ ] Produce explicit go/no-go status with owners for any carryovers
   - Exit Criteria: The release policy is green, or the final report documents a no-go decision with residual blockers and owners.

7. **Define Cross-Cutting Tracks**
   - **Evidence Freshness**: Prefer the newest artifact per gate and call out stale evidence explicitly.
   - **Contract Stability**: Backward-compatible changes only; contract drift must be cleared, not waived silently.
   - **Test Reliability**: No new flaky tests; critical launch paths must have deterministic verification.
   - **Observability**: Metrics, health, and traces must be demonstrated with real outputs.
   - **Documentation**: Document the repaired launch-gate path and final verification commands.

8. **Generate the Final Launch Checklist**
   - [ ] Claimed and verified readiness are reported separately
   - [ ] Launch-gate workflow references only commands/files that actually exist
   - [ ] Required policy/config files exist
   - [ ] Fresh arch artifact exists and passes
   - [ ] Fresh security artifact exists and passes
   - [ ] Fresh state artifact exists and passes
   - [ ] Fresh agent artifact exists and passes
   - [ ] Fresh observability artifact exists and passes
   - [ ] Smoke gate passes on the current stack
   - [ ] No unresolved cross-layer contract drift remains
   - [ ] Monitoring and dependency-aware health checks are verified
   - [ ] Kubernetes readiness is verified
   - [ ] Frontend critical screens remain evidence-backed and not merely claimed
   - [ ] L1 launch status is explicitly verified or explicitly de-scoped

9. **Present Assessment and Await Approval**
   - Present claimed readiness and verified readiness separately.
   - Show the main readiness table plus a separate L6 note.
   - Show the top 5 blockers with evidence references.
   - Present the refreshed 5-sprint plan and current checklist status.
   - Ask whether to create a new dated report under `.windsurf/plans/launch-readiness-{date}.md`.
   - Stop there. Do not create or overwrite any assessment artifact until the user explicitly approves.

10. **Create Artifacts (Only After Explicit Approval)**
    // turbo
    - Save the new report to `.windsurf/plans/launch-readiness-{date}.md`
    - Do **not** overwrite `.windsurf/plans/launch-readiness-2026-04-28.md`
    - Do **not** modify `docs/archive/2026-04-19/ARCHIVED_Phase2_Readiness_Status.md`
    - Stage changes only if the user explicitly asks for staging

## Constraints

- **Maximum 5 P0 tasks per sprint**
- **Time-boxed sprints**: 1-2 weeks each
- **Dependency-aware sequencing**: never schedule downstream work ahead of upstream gate repair
- **Measurable exit criteria**: every sprint must close with evidence, not narrative
- **Evidence-first verification**: missing or broken verification paths count against launch readiness
